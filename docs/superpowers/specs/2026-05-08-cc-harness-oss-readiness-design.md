# cc-harness v0.1.0 — Open-Source Readiness Design

**Date:** 2026-05-08
**Status:** approved (design)
**Author:** Alejandro Soto Franco

## 1. Goals and non-goals

### Goal

Make `cc-harness` install-able, discoverable, and robust enough for strangers to use without reading the source. v0.1.0 ships to GitHub with:

- a working `curl | bash` install path,
- MIT license,
- completions for bash, zsh, fish,
- a man page,
- CI on Linux and macOS,
- the today-failure-mode bug fixed (menu window dies, `Ctrl-b 0` becomes useless).

### Non-goals (explicit YAGNI)

- No rewrite. Stays bash, single primary binary.
- No new tmux features beyond what is required to keep the menu window resilient.
- No remote / multi-host orchestration.
- No GUI / TUI library. Keep `fzf`-or-numbered-menu fallback.
- No telemetry. No auto-update.
- No watchdog daemon. tmux already keeps windows alive across SSH drops; do not add a second supervision layer.

## 3. Subcommand surface

### 3.1 Full command table

| Command | Behavior |
|---|---|
| `cc-harness` (no args) | Attach (create session if needed). Existing behavior preserved. |
| `cc-harness new` | Project picker → spawn or switch (see §3.4). |
| `cc-harness new <label>` | Non-interactive spawn / switch. |
| `cc-harness new <label> --fresh` | Always spawn a new sibling window with auto-suffix (`<label>-2`, `<label>-3`, ...). |
| `cc-harness new <label> --switch` | Always switch; never spawn. Errors if no window exists. |
| `cc-harness attach [<label>]` | Open an isolated client view (grouped tmux session) focused on `<label>` or the menu. Per-terminal independent window pointer. |
| `cc-harness kill` | Pick a window, confirm, kill. |
| `cc-harness kill --tag <t>` | Filter the picker to tagged sessions. |
| `cc-harness list` | Print projects.conf as a table: `LABEL  PATH  EXISTS?  TAGS`. |
| `cc-harness list --tag <t>` | Filter table view. |
| `cc-harness list --tags` | Print unique tags with counts. |
| `cc-harness add <label> <path> [--flags '...'] [--tag t1 --tag t2]` | Append to projects.conf. Validates label, path, uniqueness. |
| `cc-harness remove <label>` | Remove row. |
| `cc-harness rename <old> <new>` | Rename projects.conf entry and the live tmux window in one shot. |
| `cc-harness tag <label> +foo -bar` | Add `#foo`, remove `#bar`. |
| `cc-harness untag <label>` | Strip all tags from a row. |
| `cc-harness logs <label>` | `tmux capture-pane -p -S -3000` for the project's window. |
| `cc-harness which <label>` | Print resolved path to stdout. |
| `cc-harness status [--json]` | Active sessions, window count + cwd, harness version, host uptime, claude binary path + version. |
| `cc-harness status --tag <t>` | Filtered status. |
| `cc-harness doctor` | Pass/fail environment checks. Exit code = number of failures. |
| `cc-harness completion <bash\|zsh\|fish>` | Emit completion script to stdout. |
| `cc-harness install [--prefix DIR] [--user]` | Write completions, man page, default config. `--user` writes to XDG / per-user dirs only (no sudo). |
| `cc-harness uninstall [--prefix DIR] [--user] [--purge]` | Reverse install. `--purge` also removes config + state. |
| `cc-harness menu` | Internal; runs the menu loop. Marked internal in help. |
| `cc-harness version`, `--version`, `-V` | Print semver + git SHA at build. |
| `cc-harness --help`, `-h` | Top-level help. |
| `cc-harness <subcmd> --help` | Per-subcommand help with examples. |

### 3.2 Global flags (apply to all subcommands)

- `--no-color` (also honors `NO_COLOR` env per the de facto standard).
- `-v` / `--verbose` — enable trace logging to `$XDG_STATE_HOME/cc-harness/cc-harness.log`.
- `-q` / `--quiet` — suppress non-error stdout.
- `--dry-run` on mutating subcommands (`add`, `remove`, `rename`, `tag`, `untag`, `install`, `uninstall`, `kill --all`).

### 3.3 Exit codes

| Code | Meaning |
|---|---|
| 0 | success |
| 1 | generic error |
| 2 | usage error (unknown subcmd, bad flag) |
| 3 | config error (parse failure, validation failure) |
| 4 | tmux not running / session not found |
| 5 | external dependency missing (`claude`, `tmux`) |
| ≥10 | reserved for `doctor` (count of failed checks) |

### 3.4 Multi-instance per project

Today's behavior dedupes by label, so each project can have at most one window ever. New behavior:

| Invocation | Behavior |
|---|---|
| `cc-harness new` (interactive picker, project has no window) | Spawn `<label>`. |
| `cc-harness new` (interactive picker, project already has a window) | Prompt: `[s]witch / [f]resh / [c]ancel`, default `s`. On `f`, spawn `<label>-2` (or next free suffix). |
| `cc-harness new <label>` (non-interactive) | Default = switch (script-safe). |
| `cc-harness new <label> --fresh` | Always spawn a new sibling. |
| `cc-harness new <label> --switch` | Always switch; error if none. |

`list`, `status`, and `kill` group windows by base label so siblings appear together.

### 3.5 Independent attachment per session

Today every claude is a tmux *window* inside one shared tmux session, so two terminals attached to the session see synchronized window selection. `cc-harness attach <label>` fixes this:

- Creates an ephemeral grouped session: `tmux new-session -d -t cc-harness -s "cc-harness-view-$$-<label>"`.
- The grouped session shares all windows with the parent but has its own current-window pointer.
- The new client immediately switches to `<label>`.
- On detach, the grouped session is destroyed (`set-option destroy-unattached on`).

Result: terminal A runs `cc-harness attach polybius`; terminal B runs `cc-harness attach pmp`; both show different claudes, both see the same underlying window set so spawns are visible to both.

`cc-harness` (no args, default attach) preserves today's "all clients share one current window" semantics so existing muscle memory works.

### 3.6 Tags and filters

Schema extension to projects.conf, all suffixes optional:

```
label = path [| flags] [#tag1 #tag2 ...]
```

Examples:

```
polybius     = ~/Polybius | --model opus     #live #trading #rust
pmp          = ~/pmp                          #math #writing
fleming      = ~/fleming                      #math #writing #paper
```

Rules:

- Tag format: `#[a-zA-Z0-9_-]+`, length 1–24.
- Multiple tags space-separated; order does not matter.
- Tags are case-insensitive at filter time.
- Parser order: split on first `=`, then on `|` if present, then take trailing `#…` tokens.

`--tag` flag added to `new`, `list`, `status`, `kill`. Multiple `--tag` flags are AND-combined; comma-list (`--tag live,trading`) is shorthand for AND. No `--any-tag` flag in v0.1; use `cc-harness list` piped through `grep` for OR.

Discovery: `cc-harness list --tags` prints unique tags with counts. Shell completion completes `--tag <value>` against the live tag set parsed from projects.conf.

## 4. Robustness

### 4.1 Self-healing

- **Menu window auto-respawn.** On every attach invocation, if the session exists but window 0 (`menu`) is missing, recreate it before switching client. This is today's failure mode; once the menu window dies it never comes back. Fix: idempotent recreate.
- **`prefix + 0` re-binding.** Bind at session level to `if-shell '! tmux list-windows -F "#W" | grep -qx menu' 'new-window -t :=0 -n menu "<self> menu"' \; select-window -t :=0`. Even if the menu died, the keystroke spawns it.
- **`prefix + N` shortcut for new project.** Bound to `run-shell '<self> new'`. One chord from any window.
- **Stale-window detection.** When attaching, check each window's pane PID; if the parent claude is dead but the window persists, rename the window to `[dead] <label>` (so it is visually flagged in the tmux window list) and offer cleanup in `kill`.

### 4.2 Preflight on attach and spawn

- Validate `claude` is on `PATH` (or `$CCH_CLAUDE` resolves) before creating a window. Today a missing binary creates an empty window; new behavior fails fast with exit 5 and the message `cc-harness: claude binary not found on PATH (or via $CCH_CLAUDE)`.
- Validate `tmux` ≥ 3.0; otherwise exit 5 with `cc-harness: requires tmux ≥ 3.0 (found <version>)`.
- Validate the project path exists and is a directory before spawning; otherwise exit 3.

### 4.3 Error handling

- Replace silent `|| true` patterns with explicit error reporting at the user-facing layer.
- All user-facing errors go to stderr with a `cc-harness: <subcmd>: <reason>` prefix.
- Non-zero exits per the §3.3 table.
- Add `IFS=$'\n\t'` for safer word-splitting around config parsing (`set -euo pipefail` already on).

### 4.4 Single-instance protection

Lockfile at `$XDG_RUNTIME_DIR/cc-harness.lock` (fallback `/tmp/cc-harness-$UID.lock`) around `add`, `remove`, `rename`, `tag`, `untag`, `install`, `uninstall` to prevent concurrent config mutations corrupting projects.conf.

### 4.5 Logging

`-v` / `--verbose` enables trace-style logging to `$XDG_STATE_HOME/cc-harness/cc-harness.log`. Rotated at 1 MiB, keep 3. Default off, no log file created. `cc-harness status` reports the log path if present.

### 4.6 Signal handling

Trap `SIGINT` / `SIGTERM` in the menu loop to redraw cleanly instead of exiting mid-fzf with a broken terminal.

## 5. Configuration and XDG migration

### 5.1 Path resolution

Priority order, first hit wins:

| Priority | Config path | State path |
|---|---|---|
| 1 | `$CCH_HOME/projects.conf` (if `CCH_HOME` set) | `$CCH_HOME/` |
| 2 | `$XDG_CONFIG_HOME/cc-harness/projects.conf` | `$XDG_STATE_HOME/cc-harness/` |
| 3 | `~/.config/cc-harness/projects.conf` | `~/.local/state/cc-harness/` |
| 4 | `~/cc-harness/projects.conf` (legacy fallback) | `~/cc-harness/` (legacy fallback) |

`CCH_HOME` is preserved as an explicit override. The legacy fallback exists only so existing installs do not break before migration runs.

### 5.2 Migration trigger

On any invocation, after path resolution, if all three hold:

- the resolved config came from priority 4 (legacy `~/cc-harness/`),
- the canonical XDG config path (priority 2; falls back to priority 3 when `$XDG_CONFIG_HOME` is unset) does not yet contain a `projects.conf`,
- `CCH_HOME` is not explicitly set,

then run a one-shot migration:

1. `mkdir -p` the XDG config dir.
2. `cp -p ~/cc-harness/projects.conf $XDG_CONFIG_HOME/cc-harness/projects.conf`.
3. Touch a marker file `$XDG_STATE_HOME/cc-harness/.migrated-from-legacy`.
4. Print to stderr (one-time, gated by the marker):
   ```
   cc-harness: migrated config from ~/cc-harness/ to ~/.config/cc-harness/
   cc-harness: legacy ~/cc-harness/ left in place; remove when ready
   ```
5. Continue with the new path.

Legacy directory is **not** deleted automatically. `cc-harness doctor` reports its presence and suggests removal once the user has confirmed everything works.

### 5.3 State files

- `$XDG_STATE_HOME/cc-harness/cc-harness.log` — debug log (only when `-v`).
- `$XDG_STATE_HOME/cc-harness/.migrated-from-legacy` — migration marker.
- `$XDG_RUNTIME_DIR/cc-harness.lock` — ephemeral lockfile.

### 5.4 Config format

- Preserves the existing `label = path` format. Schema extends to `label = path [| flags] [#tag1 #tag2 ...]`; all suffixes optional.
- `add`, `remove`, `rename`, `tag`, `untag` preserve comments and ordering by editing in place via temp-file + atomic `mv`.
- `~` expansion happens at parse time, not on write; the file remains portable across users.

### 5.5 Validation rules

- Label: `^[a-zA-Z0-9_-]+$`, length 1–32.
- Path: must resolve to an existing directory after `~` expansion.
- No duplicate labels.
- No leading/trailing whitespace on values.
- Tag format per §3.6.
- Errors point to the offending line number.

### 5.6 Environment variables

| Name | Purpose | Default |
|---|---|---|
| `CCH_HOME` | Override config + state dir | unset (use XDG) |
| `CCH_FLAGS` | Flags passed to `claude` on every spawn | `--dangerously-skip-permissions` |
| `CCH_CLAUDE` | Path to the claude binary | `claude` (on `PATH`) |
| `CCH_SESSION` | tmux session name | `cc-harness` |
| `NO_COLOR` | Disable ANSI color output | unset |

## 6. Packaging

All channels read from a single source of truth: the GitHub release tarball (script + man source + completions/, generated at tag-time by CI).

### 6.1 `install.sh` (curl | bash, primary)

```
curl -fsSL https://raw.githubusercontent.com/<user>/cc-harness/main/install.sh | bash
```

Behavior:

- Detects OS (linux/macos) and arch.
- Hits GitHub Releases API for latest tag (or `CC_HARNESS_VERSION=v0.1.0` env override).
- Downloads tarball + checksum to `$TMPDIR`, verifies SHA-256.
- Installs to `--prefix` (default: `/usr/local` if writable, else `~/.local`).
- Drops completions into the right shell-specific dir.
- Idempotent: re-run = upgrade.
- `--uninstall` flag delegates to `cc-harness uninstall`.
- Shellcheck-clean and audited; a `curl | bash` install will be read by skeptics.

### 6.2 `Makefile`

Targets, all DESTDIR-aware so packagers can stage:

```
make                   # default: build (regenerate completions + man from script)
make completions       # cc-harness completion <shell> > completions/cc-harness.<ext>
make man               # pandoc docs/cc-harness.1.md → docs/cc-harness.1
make install           # install bin, man, completions, default config
make uninstall         # reverse install
make test              # bats tests
make lint              # shellcheck
make deb               # build .deb via fpm or dpkg-deb
make rpm               # build .rpm via fpm or rpmbuild
make dist              # tarball + checksums for release
make clean
```

Override variables: `PREFIX` (default `/usr/local`), `DESTDIR` (default empty), `MANDIR` (`$PREFIX/share/man`), `BASH_COMPLETION_DIR`, `ZSH_COMPLETION_DIR`, `FISH_COMPLETION_DIR`.

### 6.3 Homebrew tap

Separate repo: `<user>/homebrew-cc-harness`. Single file at `Formula/cc-harness.rb`:

```ruby
class CcHarness < Formula
  desc "Multi-session Claude Code launcher backed by tmux"
  homepage "https://github.com/<user>/cc-harness"
  url "https://github.com/<user>/cc-harness/releases/download/v0.1.0/cc-harness-0.1.0.tar.gz"
  sha256 "..."
  license "MIT"
  depends_on "tmux"
  depends_on "fzf" => :recommended

  def install
    system "make", "install", "PREFIX=#{prefix}"
    bash_completion.install "completions/cc-harness.bash"
    zsh_completion.install  "completions/cc-harness.zsh" => "_cc-harness"
    fish_completion.install "completions/cc-harness.fish"
    man1.install "docs/cc-harness.1"
  end

  test do
    assert_match "cc-harness", shell_output("#{bin}/cc-harness --version")
  end
end
```

CI in the main repo updates the formula's `url` and `sha256` on tag release via PR to the tap repo.

### 7.2 Test isolation

Every test uses `tmux -L cc-harness-test-$BATS_TEST_NUMBER` (per-test socket; parallel-safe) and a per-test `$HOME` under `$BATS_TEST_TMPDIR`. `claude` is stubbed with a script that `sleep 9999`s, so spawned windows do not actually run Claude in CI.

### 7.3 Coverage target

Every subcommand has at least one happy-path test and one error-path test. Today's failure mode (menu window dies) gets a regression test in `menu_respawn.bats`. No coverage % enforced; bash coverage tooling is fragile.

### 7.4 Lint

`.shellcheckrc` sets `external-sources=true`, `shell=bash`. `make lint` runs:

```
shellcheck -x bin/cc-harness install.sh tests/test_helper.bash
```

CI fails on any warning. Disables documented inline (`# shellcheck disable=SCxxxx — reason`).

### 7.5 Format

`shfmt -i 4 -ci -bn -d` enforces 4-space indent, switch-case indent, binary ops at line start. Checked in CI.

### 7.6 GitHub Actions CI

`.github/workflows/ci.yml` runs three jobs:

- `lint` — shellcheck + shfmt on Ubuntu.
- `test` — bats matrix on `{ubuntu-latest, macos-latest} × {bash 3.2, bash 5.2}` so portability regressions show up immediately. macOS system bash is 3.2 (catches associative-array, `mapfile`, `[[ -v ]]` regressions). macOS + Homebrew bash 5.2 catches BSD-vs-GNU userspace issues (`sed -i`, `readlink -f`, `date`).
- `release` — tag-driven; runs `make dist deb rpm`, attaches artifacts to the GitHub Release, triggers Copr build via webhook, opens a PR to bump the Homebrew formula.

If a feature genuinely cannot be made bash-3.2 portable, gate it behind a runtime check + clear error message rather than silently breaking.

### 7.7 Pre-commit (optional, contributor-side)

`.pre-commit-config.yaml` configures shellcheck, shfmt, and a fast bats unit-tests hook. CI is authoritative; pre-commit is convenience.

## 10. Deferred to v0.2+

- Pre/post-spawn hooks at `$XDG_CONFIG_HOME/cc-harness/hooks/{pre,post}-spawn`.
- Manual `self-update` subcommand (signature verification, release-feed dependency).
- Nixpkgs PR.
- Ubuntu PPA.
- Official Homebrew core / Fedora / Debian — gated on adoption metrics, not engineering.

## 11. Explicitly rejected

- `cc-harness send <label> <text>` — `tmux send-keys` wrapper. Looks innocent; becomes a remote-command-execution vector.
- Project state in window names (`[busy]`, `[idle 5m]`) — requires polling claude internals not exposed.
- Session save/restore — `tmux-resurrect` does this well; reference in README instead of reimplementing.
- Telemetry, auto-update, watchdog daemon — see §1 non-goals.
